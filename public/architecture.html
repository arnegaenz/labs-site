01 — Product Overview

HearZ is a curated audio feed app that reads the best articles from the web aloud using AI-generated voices (OpenAI TTS). Open the app, tap play — like a radio station for articles. The primary audience is elderly users. Every UX decision filters through: "would a 75-year-old understand this without help?" Accessibility-first, zero friction to value.

Listen

OpenAI TTS reads every article with premium AI voices, teleprompter-style word highlighting, and auto-scroll. This is the entire product.

Vote

One article at a time, like Tinder. Skip, Listen, or Love. Three big buttons. No scrolling, no choice paralysis. Feed gets smarter with every vote.

Personalize

AI fingerprinting + vote-based taste profile + source preferences. The ranking algorithm learns what you love and surfaces more of it.

Product Vision

  • Zero friction: Download → Open → It's already playing
  • No account required, no onboarding wall — just works
  • Ad-supported free tier, Pro upgrade for power users
  • Audio ads between articles (the big money — 5-10x banner CPMs)

Naming

  • Product: HearZ (always)
  • Repo: HearZee (legacy, do not rename)
  • Bundle ID: com.skoobilabs.hearzee
  • Internal persona: "HearZer" (never in app UI)
  • Studio: SkoobiLabs
  • Tagline: "Your feed, out loud."

08 — TTS System — The Core Feature

HearZ uses AI to read every article aloud. This is the entire product. Without this, there is no product.

OpenAI TTS generates premium AI voices. Whisper provides word-level timestamps. The teleprompter highlights each word in real-time as it's spoken. Audio is cached in R2 so the same text is never generated twice.

Step-by-Step TTS Flow

Generation (Backend)

  1. Article text split into sections by SectionSplitter.ts
  2. Each section POSTed to backend /api/audio
  3. Backend calls OpenAI TTS API to generate MP3
  4. Backend calls OpenAI Whisper for word-level timestamps
  5. MP3 cached in R2 bucket with content-hash key
  6. Returns: MP3 URL + array of { word, startTime, endTime }

Playback (Client)

  1. App plays MP3 via expo-audio
  2. Teleprompter highlights words as timestamps are reached
  3. Text scrolls in sync with the voice
  4. Next 2 sections pre-fetched for gapless playback
  5. When all sections finish → audio ad may play → next article

TTS Engine Selection

Streamed (DEFAULT)StreamedTTS.tsOpenAI TTS API via backendPremium AI voices~$0.015/1K charsLive
Native (fallback)NativeTTS.tsiOS AVSpeechSynthesizerRoboticFreeSafety net

Available voices: alloy, echo, fable, onyx, nova, shimmer (OpenAI voice IDs). Setting: ttsEngine in MMKV, default is 'streamed'.

AudioService.ts singleton

Main orchestrator. Manages play, pause, resume, stop, seek, speed. Section-loop architecture: iterates through sections sequentially. Prefetches next 2 sections. Coordinates with AudioMutex for concurrency safety. Emits state via subscriber pattern.

play() pause() resume() stop() seek() setSpeed()

AudioMutex.ts

Prevents two things playing at once (e.g., preview + full playback). Uses generation-based token invalidation — if a new play() starts, the old one's token becomes invalid and it stops gracefully.

NowPlayingService.ts

iOS lock screen + Control Center integration. Runs a silent expo-audio player to claim the audio session (because AVSpeechSynthesizer doesn't register with MPNowPlayingInfoCenter). Proxies remote commands to AudioService.

PreviewService.ts

Short 5-10 second previews of article title + first sentence. Pre-warms audio on Home screen load via warmup(). Auto-plays when a new vote card appears. Separate from full playback — uses AudioMutex to coordinate.

10 — Vote-Based Home Screen

The Home screen is NOT a scrollable feed. It shows ONE article at a time, like Tinder for articles. Designed for elderly users — one decision at a time, no scrolling, no choice paralysis.

How It Works

  1. On load: getRankedArticles() fetches 50 articles, scored and sorted
  2. Top-ranked article displayed as large card with image, title, source, duration
  3. Article title + first sentence auto-plays as preview (AI voice, ~5-10s)
  4. User sees three big buttons (60pt+ tap targets):

Skip

Card animates left. Topics -2, source -1. Recorded in vote_history.

Listen

Opens full player. Passive topic boost +1 on play.

Love

Card animates right. Topics +3, source +1. Opens player.

Why This Design

  • Elderly users — one decision at a time
  • Three big buttons — universal icons, 60pt+ targets
  • Audio preview auto-plays — hear what it's about before deciding
  • Feels like radio — "Here's what's next, want to hear it?"
  • Feed gets smarter — every vote teaches the algorithm

Also on Home Screen

  • [Aa] button (top right) — TextSizePopover
  • Continue Listening section below card
  • Banner ad for free-tier users
  • Queue counter: "12 articles in your feed"

11 — Smart Ranking Algorithm

The ranking algorithm determines what article the user sees next. Implemented in src/services/db/articles.ts → getRankedArticles().

Scoring Pipeline

  1. Fetch: All successful articles (3x limit for headroom)
  2. Exclude: Articles already voted on (from vote_history table)
  3. Deduplicate: Near-identical articles (>70% title word overlap)
  4. Score each article:
FactorWeightSource
Topic scoreSum of taste_profile weights for matching topicstaste_profile table
Refined topic scoreFingerprint topics_refined matched at 0.5x weightFingerprint JSON
Source scoresource_preferences weight × 2 (source matters more)source_preferences table
Fingerprint bonus+1 simple complexity, +0.5 positive sentiment, +0.5 inspirational toneFingerprint JSON
Recency bonusUp to +10 for brand new, decays linearly over 60 hourspublished_at

Post-Scoring

Source interleaving: max 2 from same source in any 6-item window. Zero-vote fallback: algorithm works well even with no votes — recency and source diversity provide a good default. Votes make it progressively better.

Fingerprint Fields

  • sentiment — positive, negative, neutral
  • complexity — simple, moderate, complex
  • format — news, opinion, how-to, listicle, etc.
  • tone — informational, inspirational, critical, humorous
  • topics_refined — precise topics replacing coarse source-level ones
  • entities — people, companies, places mentioned
  • length_category — short (<450), medium (<1200), long (1200+) words

Implementation Details

  • File: backend/src/services/fingerprint/classifier.ts
  • Runs: every 15 min cron, 10 articles per batch
  • Cost: ~$0.001 per article
  • Secret: ANTHROPIC_API_KEY on Worker
  • Failure handling: stores '{}' to prevent retrying
  • Topic upgrade: coarse source topics (e.g., "technology") replaced with Claude's refined topics (e.g., "artificial intelligence", "self-driving cars")

13 — Taste Profile & Source Preferences

Every user interaction updates the taste profile and source preferences, personalizing the feed over time.

Taste Profile (taste_profile table)

Per-topic weights that drive article ranking.

ActionWeight
Play article+1 topic weight
Complete article+2 topic weight
Vote up (Love)+3 topic weight
Vote down (Skip)-2 topic weight

Weight never goes below 0. Visualized as animated bubble chart on Profile tab.

Source Preferences (source_preferences table)

Per-source weights that track user affinity for specific publications.

ActionWeight
Vote up (Love)+1 source weight
Vote down (Skip)-1 source weight

Tracks up_count and down_count separately for pattern detection. "Always downvotes Fox News" = source preference, not topic preference.

Vote History (vote_history table)

Full thumbs up/down log with article_id, source, topics, vote_type, timestamp. Used by ranking to exclude already-voted articles and for pattern analysis.

Behavior Signals (useBehaviorTracking)

Playback events: play_start, pause, resume, skip_forward/backward, seek, speed_change, save, share, vote_up, vote_down. Synced to backend every 5 minutes + on app foreground.

AudioAdService.ts

  • Frequency: every 3rd article transition (AD_FREQUENCY = 3)
  • Alternating cycle tracked in AsyncStorage:
  • Cycle 0: Network ad slot — currently TTS placeholder, future real ad network
  • Cycle 1: Self-promo — drives Pro upgrades ("Upgrade to HearZ Pro for $4.99/mo")
  • Pattern: Article 1 → 2 → 3 → [Network ad] → 4 → 5 → 6 → [Self-promo] → repeat
  • Skip available after 5 seconds, auto-stops at 15 seconds max
  • Pro users hear ZERO ads — checks entitlements first

AudioAdOverlay Component

  • "AD" badge displayed during audio ad playback
  • Skip button appears after 5-second delay
  • All events tracked: impressions, skips, elapsed time at skip
  • Renders in tab layout alongside MiniPlayer

Future: Real Ad Networks

getNetworkAd() currently returns TTS-spoken tips. At 1K+ users, integrate Google Ad Manager or AdsWizz for real paid audio ads via VAST tags.

25 — Deep Links & Referrals

Share Flow

  1. User taps Share on article (NowPlaying screen)
  2. createShareUrl(id, title) generates URL
  3. URL: https://hearz.app/article/{id}?ref={code}
  4. Referral code = first 8 chars of device ID
  5. Native share sheet opens (title + URL)

Capture Flow

  1. App opens from deep link
  2. Linking.getInitialURL() in root layout
  3. captureReferral(url) extracts ref param
  4. First referral stored in AsyncStorage
  5. Referral count tracked per device

URL scheme: hearz:// (registered in app.json). Web fallback: hearz.app domain not yet purchased

29 — Security Model

Client-Side

  • MMKV encrypted storage for settings/tokens
  • Supabase session tokens in secure storage
  • No secrets in client code
  • Device ID for anonymous identification
  • Entitlements verified locally + synced from cloud
  • Analytics batched locally before server sync

Server-Side

  • CORS enabled (specific origins)
  • API keys stored as Worker secrets (OPENAI_API_KEY, ANTHROPIC_API_KEY)
  • Admin routes protected
  • Rate limiting via KV
  • No PII stored in D1 (device ID only)
  • R2 audio keyed by content hash (no user data)
